The part of backend engineering that nobody finds interesting is the setup — and it's also the part where most services end up subtly broken. The logger doesn't redact PII. The shutdown handler doesn't wait for in-flight requests. The HTTP client doesn't have a circuit breaker. The config loading works in development but panics when an env var is missing in production. None of these are hard problems individually, but setting them up correctly takes time, and under deadline pressure they get done just well enough to ship.

archway doesn't just scaffold files. It generates these patterns correctly — the way you'd write them if you had a full sprint to do it right.

## What archway does for backend engineers

- Generates structured logging with `slog` and PII redaction built in — sensitive fields are scrubbed before they hit your log pipeline
- Wires graceful shutdown via `lifecycle.App` with ordered component shutdown and configurable timeouts — no more `SIGTERM` killing in-flight requests
- Scaffolds the bootstrap pattern so every dependency is injected, not instantiated — services are testable without mocks from day one
- Sets up OpenTelemetry traces and metrics with OTLP export — distributed tracing across services without manual instrumentation

## The problem it solves

Backend engineers who build services from scratch face the same problem repeatedly: the infrastructure setup is complex, the correct patterns are well-known but tedious to implement, and the cost of getting them wrong shows up later in production — not during development. RFC 7807 Problem Detail errors are the right format for REST APIs, but most services use ad-hoc JSON structures because there was no time to implement it properly. OpenTelemetry is the right observability standard, but wiring it correctly into a Chi middleware chain requires knowing how it works.

archway encodes these patterns as capabilities. When you add `http-api`, you get RFC 7807 errors and a correctly-wired middleware chain. When you add `observability`, you get OpenTelemetry with trace propagation. You don't have to know the details of how each is implemented — you just have to know you want it.

## How it works

```bash
# What a backend engineer reaches for on day one of a new service
archway new my-service --arch layered --cap platform,bootstrap,http-api,health,request-id,validation,postgres,migrations,uuid,docker
```

The result is a compilable Go service with Chi router, pgx PostgreSQL, golang-migrate, request ID propagation, input validation with go-playground/validator, `slog` logging, and Docker Compose. The handler layer, service layer, and repository layer are already separated, and the dependency rules between them are encoded in `archway.yaml`.

<Aside type="tip">
The capability warning system flags dangerous combinations before you commit. `retry` without `idempotency` is a production incident waiting to happen — retrying a payment endpoint that creates a charge will double-charge the customer. archway surfaces this at scaffold time, not in a post-mortem.
</Aside>
