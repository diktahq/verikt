Platform teams are responsible for the standards that no individual service team enforces. You write the architecture decision records, you run the internal developer platform, you define what "production-ready" means. The problem is enforcement. You can document that every service should use hexagonal architecture, have health endpoints, and not call external services without circuit breakers — but documentation doesn't stop a developer under deadline from importing an adapter package directly from the domain layer. You only find out months later, in a code review, or worse, in an incident.

`archway check` is the enforcement layer you've been missing.

## What archway does for platform teams

- Runs 11 AST-based architecture detectors in CI as a single binary, no external dependencies — add it to any service's pipeline in one step
- Catches dependency violations deterministically: domain code importing adapter code, service layer importing handler layer, bounded context A importing bounded context B's internals
- Validates that declared capabilities are correctly implemented — if the service claims hexagonal architecture, the actual import graph is verified against the rules
- Integrates with the scaffolding layer — when `archway new` scaffolds a service, it writes `archway.yaml` with the architecture rules, and `archway check` reads them

## The problem it solves

Architecture governance at scale has a fundamental problem: the gap between what was decided and what was built grows over time, and nobody has time to audit it. The platform team publishes standards. Engineering teams build services under pressure. Code reviews catch obvious violations but miss subtle ones — a service layer that imports directly from the infrastructure layer, a domain entity that depends on a configuration struct. These violations are individually small but collectively they erode the architecture until the documented patterns no longer describe the actual code.

The second problem is that manual enforcement doesn't scale. With 5 services, you can review each one. With 20, you can't. The platform team ends up choosing between enforcing standards and shipping platform features — and standards usually lose.

`archway check` runs in CI. It's deterministic, fast, and reports exactly which files violate which rules. When it's in every service's pipeline, the platform team isn't the last line of defense — the CI gate is.

## How it works

```bash
# The command platform teams add to every service's CI pipeline
archway check --path ./my-service
```

Each service's `archway.yaml` declares its architecture and the rules that follow from it. `archway check` reads that file and validates the actual codebase against those rules. It exits non-zero on violations, which fails the CI pipeline. No violation goes undetected because no service can ship without passing the check.

<Aside type="tip">
The most effective deployment pattern: define a baseline capability set at the org level — `platform,bootstrap,http-api,health,request-id,observability,ci-github` — and use `archway new` with `--no-wizard` in your internal service scaffolding tooling. Every service gets the baseline. `archway check` verifies it stays that way.
</Aside>
