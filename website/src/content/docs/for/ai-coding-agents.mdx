AI coding agents are productive until they aren't. They write correct Go syntax, they know common patterns, they can implement a feature in minutes. The problem is architectural context. When Claude or Cursor opens a file in your hexagonal service, it doesn't know that domain code can't import from adapter code, that configuration structs live in the platform layer, that the repository interface belongs in `port/outbound.go` not in the adapter. It invents conventions based on what it's seen in training data — and those conventions are usually different from yours.

The result is architecturally coherent code that violates your service's rules. The agent is confident. The code compiles. And then `archway check` fails in CI with 8 dependency violations.

`archway guide` prevents this.

## What archway does for AI coding agents

- Generates instruction files from your `archway.yaml` that describe your exact architecture, capability set, and the rules that follow from both — not generic Go advice, your service's rules
- Outputs to the format each agent reads: `.claude/rules/archway.md` for Claude Code, `.cursorrules` for Cursor, `.github/copilot-instructions.md` for Copilot, `.windsurfrules` for Windsurf
- Includes capability-specific guidance — if your service has `retry` + `idempotency`, the instruction file explains why both are required and how they relate
- Runs automatically as a post-scaffold hook after `archway new`, so every new service starts with agent context already configured
- Re-run `archway guide` after adding capabilities to keep the instruction files current

## The problem it solves

Context loss is the central failure mode of AI-assisted development. An agent working on file 50 of a service has lost the architectural decisions made when the project started. It doesn't remember that the team chose hexagonal architecture, that the `platform` capability owns config loading, that all outbound dependencies are abstracted behind port interfaces. Without that context, it defaults to patterns from its training data — which may be internally consistent but are wrong for your codebase.

The standard workaround is CLAUDE.md files and manual instruction writing. But those instructions go stale. They're written once, cover the obvious things, and don't update when you add a new capability or refactor a layer. The agent works with the instructions from six weeks ago while the codebase has moved on.

`archway guide` generates instructions from the authoritative source — `archway.yaml` — so they're always current. Adding `circuit-breaker` to your service? Re-run `archway guide` and the agent now knows the circuit breaker lives in the adapter layer, wraps outbound HTTP calls, and should be paired with `retry` in the http-client setup.

## How it works

```bash
# Run after scaffolding or after adding capabilities
archway guide
```

archway reads your `archway.yaml`, resolves your architecture pattern and installed capabilities, and generates instruction files for every configured agent target. For a hexagonal Go service with `http-api`, `postgres`, `circuit-breaker`, and `retry`, the generated `.claude/rules/archway.md` will include:

- Layer boundaries and which packages belong where
- Dependency rules (domain can't import from service, adapters can't import from each other)
- Capability-specific patterns (where the circuit breaker wraps, how the repository interface is structured)
- Smart suggestion context (why retry and idempotency are paired, why http-api suggests health and request-id)
- Anti-pattern warnings (what `archway check` will catch and reject)

The agent reads this file before every session. It knows your architecture. It writes code that fits.
